# Development Plan

## 项目目标

做一个基于 Web 技术的多人炸弹人 App。

核心方向：

- Phaser 客户端
- Colyseus 多人服务端
- Capacitor 打包 Android / iOS
- 后续接入硬件即时反馈

## 基本假设

- 第一版先做 2-4 人实时对战。
- 先保证房间、玩法、同步稳定，再做美术和运营功能。
- 服务端优先做权威结算，避免客户端作弊和状态不一致。
- 硬件反馈先做事件层，不一开始绑定具体设备。

## 当前状态

已完成：

- Phaser + Colyseus 基础项目
- `bomberman_room` 房间
- 玩家加入房间
- 玩家移动同步
- 服务端固定帧处理输入
- 阶段 1 核心玩法 MVP
- 阶段 2 多人房间大厅
- 阶段 3 道具和局内节奏
- 阶段 4 移动端操作

验证方式：

```bash
npm run build
npm --prefix server test
```

## 阶段 1：核心玩法 MVP

目标：先让炸弹人玩法跑起来。

开发内容：

- 地图格子系统
- 固定墙体
- 可破坏障碍物
- 玩家出生点
- 放炸弹
- 炸弹倒计时
- 十字爆炸
- 爆炸命中玩家
- 死亡和回合结束

验收标准：

- 2 个玩家能进入同一房间。
- 玩家能移动和放炸弹。
- 炸弹能按格子爆炸。
- 爆炸能破坏障碍物。
- 玩家被炸中后死亡。
- 一方存活时能结束回合。
- `npm run build` 通过。

## 阶段 2：多人房间大厅

目标：把随机进房间改成可控房间。

开发内容：

- 创建房间
- 输入房间号加入
- 房间列表
- 玩家昵称
- 房主
- 准备状态
- 房主开始游戏
- 退出房间
- 房间人数上限

验收标准：

- 玩家可以创建私人房间。
- 其他玩家可以通过房间号加入。
- 所有人准备后，房主可以开始。
- 游戏中离开不会导致服务端报错。
- 房间人数超过上限时不能加入。

## 阶段 3：道具和局内节奏

目标：让对战有变化。

开发内容：

- 炸弹数量道具
- 爆炸范围道具
- 移速道具
- 护盾或复活道具
- 道具随机掉落
- 回合计时
- 多局积分
- 胜负结算页

验收标准：

- 道具只由服务端生成和结算。
- 玩家拾取后属性同步给所有客户端。
- 一局结束后能进入下一局。
- 多局分数正确累计。

## 阶段 4：移动端操作

目标：让游戏适合手机 App。

开发内容：

- 虚拟摇杆
- 放炸弹按钮
- 技能按钮预留
- 横屏适配
- 安全区适配
- 触摸反馈
- 基础音效开关

验收标准：

- 手机浏览器可以操作。
- Android Chrome 手感可接受。
- iOS Safari 基础显示正常。
- 横屏下 UI 不遮挡战斗区域。

## 阶段 5：硬件反馈事件层

目标：先把游戏事件和硬件解耦。

开发内容：

- 新增 `FeedbackEvent` 类型
- 炸弹放置事件
- 爆炸预警事件
- 被炸中事件
- 死亡事件
- 捡道具事件
- 胜利 / 失败事件
- 客户端统一分发反馈事件

验收标准：

- 没有硬件时游戏正常运行。
- 打开调试模式能看到反馈事件日志。
- 后续接 BLE 时不需要改核心玩法。

## 阶段 6：Capacitor App

目标：打包成 Android / iOS App。

开发内容：

- 接入 Capacitor
- Android 工程
- iOS 工程
- 本地网络配置
- App 图标和启动图
- 构建脚本
- 测试包输出

验收标准：

- Android 能安装运行。
- iOS 模拟器能运行。
- App 能连接测试服。
- WebSocket 正常。

## 阶段 7：BLE 硬件接入

目标：把反馈事件接到真实硬件。

开发内容：

- 选择 BLE 插件
- 扫描设备
- 连接设备
- 断线重连
- 发送反馈指令
- 设备状态显示
- 反馈强度配置

验收标准：

- Android 可以连接设备。
- iOS 可以连接设备。
- 受击、爆炸、胜负能触发真实反馈。
- 设备断开后游戏不崩溃。

## 阶段 8：体验打磨

目标：从 Demo 变成可试玩产品。

开发内容：

- 主菜单
- 角色选择
- 地图选择
- 简单美术替换
- 音效
- 动画
- 网络延迟提示
- 断线提示
- 重连逻辑

验收标准：

- 新玩家能自己完成开房和对战。
- 弱网下有明确提示。
- 主要按钮和文字不遮挡。
- 一局体验完整。

## 阶段 9：上线前准备

目标：准备内测。

开发内容：

- 测试服部署
- 日志收集
- 错误上报
- 基础反作弊
- 版本号
- 隐私协议
- 用户协议
- 测试包分发

验收标准：

- 测试人员能安装 App。
- 服务端能稳定跑多个房间。
- 能定位客户端和服务端错误。
- 有基本隐私和权限说明。

## 优先级

当前最高优先级：

1. 阶段 5：硬件反馈事件层
2. 阶段 6：Capacitor App
3. 阶段 7：BLE 硬件接入
4. 阶段 8：体验打磨

暂时不做：

- 账号系统
- 充值系统
- 排位赛
- 好友系统
- 大规模匹配
- 复杂皮肤系统

## 下一步任务

建议下一次从阶段 5 开始：

1. 定义 `FeedbackEvent` 类型。
2. 在服务端关键玩法事件处生成反馈事件。
3. 客户端统一接收和分发反馈事件。
4. 增加调试日志开关。
5. 保持没有硬件时游戏正常运行。
