# Development Plan

## 项目目标

做一个基于 Web 技术的多人炸弹人 App。

核心方向：

- Phaser 客户端
- Colyseus 多人服务端
- Capacitor 打包 Android / iOS
- 后续接入硬件即时反馈

## 基本假设

- 第一版先做 2-4 人实时对战。
- 先保证房间、玩法、同步稳定，再做美术和运营功能。
- 服务端优先做权威结算，避免客户端作弊和状态不一致。
- 用户、登录注册、战绩和排行榜统一接入 MySQL。
- 服务端通过 Prisma 管理数据库表结构、迁移和查询。
- 对局过程中仍以内存状态为主，对局结束后一次性写入数据库。
- 硬件反馈先做事件层，不一开始绑定具体设备。

## 开发执行规范

默认按轻量、直接、可验证的方式推进：

- 先说明当前假设、不确定点和边界条件。
- 优先选择能解决当前问题的最简单方案。
- 只改和当前需求直接相关的代码。
- 不做顺手重构，不扩大改动范围。
- 完成前必须做可验证检查。

按场景使用加强流程：

- 新功能、行为变更、界面设计：先做简短方案梳理。
- 多步骤任务：先拆阶段，再逐步验证。
- Bug 修复：先定位原因，再做最小修复。
- 准备交付前：必须做完成前验证。
- 代码评审：优先指出风险、缺陷和缺少的测试。

代码要求：

- 核心业务逻辑、主要方法步骤、关键规则需要补充中文注释。
- 优先使用 `apply_patch` 修改文件。
- 安装必要依赖时优先安装到 D 盘。
- 不回滚用户已有改动，遇到冲突先基于现状处理。

提交流程：

- 提交前先运行必要检查。
- 提交信息使用 Conventional Commits。
- `type` 和 `scope` 使用英文。
- `subject` 和 `body` 使用中文。
- 示例：`feat(room): 增强房间流程和开局倒计时`

常用验证命令：

```bash
npm run build
npm --prefix server test
```

## 阶段 6：房间流程增强

目标：让开房、等人、开局流程更完整。

开发内容：

- 创建房间时选择最大人数
- 创建房间时设置房间密码
- 加入加密房间时输入密码
- 房间列表展示公开 / 私密状态
- 房主转让
- 踢出玩家
- 准备状态强化展示
- 开局倒计时
- 房间内地图展示更清楚
- 房主离开后自动转让房主
- 房间状态同步到所有玩家

验收标准：

- 创建房间时可选择 2 / 3 / 4 人。
- 超过人数上限时不能加入。
- 房主离开后能正确转让。
- 房主可以管理房间成员。
- 非房主不能踢人或转让房主。
- 私人房间可以防止陌生人直接加入。
- 密码错误或未输入密码时不能加入私人房间。
- 所有人准备后有倒计时再开局。
- 倒计时中有人取消准备或离开时会取消开局。
- 房间状态所有玩家同步一致。

建议实现顺序：

1. 服务端补齐人数、密码、踢人、转让房主和倒计时规则。
2. 服务端补充房间流程测试。
3. 客户端更新创建房间、加入房间和房间等待页 UI。
4. 浏览器验证创建房间、加入房间、踢人、转让和倒计时。
5. 更新本文档当前状态。

## 阶段 12：账号系统和数据持久化

目标：把本地用户信息、登录注册、对局结果和战绩接入数据库。

技术选择：

- 数据库：MySQL 8
- ORM：Prisma
- 密码加密：bcrypt
- 登录凭证：JWT
- 数据写入策略：对局中不频繁写库，对局结束后统一落库

开发内容：

- 安装并配置 Prisma
- 新增 MySQL 连接配置
- 新增数据库迁移脚本
- 新增用户表 `users`
- 新增对局表 `matches`
- 新增对局玩家表 `match_players`
- 新增注册接口
- 新增登录接口
- 新增登录态校验中间件
- 本地个人信息迁移到服务端用户资料
- 本地战绩迁移到服务端战绩查询
- 对局结束后保存对局结果
- 新增个人战绩查询接口
- 新增排行榜查询接口
- 客户端接入登录 / 注册页面
- 客户端保存登录态
- 未登录时限制进入联机功能

建议表结构：

```txt
users
- id
- username
- password_hash
- nickname
- avatar
- character_key
- created_at
- updated_at

matches
- id
- room_id
- map_key
- status
- started_at
- ended_at
- winner_user_id
- raw_data
- created_at

match_players
- id
- match_id
- user_id
- nickname
- rank
- score
- kills
- deaths
- survived_seconds
- created_at
```

验收标准：

- 用户可以注册账号。
- 用户可以登录账号。
- 密码不会明文入库。
- 登录后能进入联机功能。
- 未登录用户不能直接进入需要账号的联机流程。
- 对局结束后能写入 `matches` 和 `match_players`。
- 个人信息从数据库读取和保存。
- 历史战绩从数据库查询。
- 排行榜可以按胜场、积分或胜率查询。
- 数据库连接失败时服务端有明确错误日志。
- `npm run build` 通过。
- `npm --prefix server test` 通过。

建议实现顺序：

1. 配置 MySQL、Prisma 和环境变量。
2. 设计并生成 `users`、`matches`、`match_players` 表。
3. 实现注册、登录和登录态校验。
4. 客户端接入登录 / 注册流程。
5. 对局结束时保存比赛结果。
6. 接入个人信息、历史战绩和排行榜查询。
7. 补充服务端测试。
8. 运行 `npm run build`、`npm --prefix server test`。
